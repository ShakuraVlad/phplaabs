# Лабораторная работа №6. Обработка и валидация форм

## Тема проекта: Дневник настроения

## Модель данных

| Поле          | Тип            | Описание                                  |
|----------------|----------------|--------------------------------------------|
| title          | string         | Заголовок записи                          |
| entry_date     | date           | Дата записи                               |
| mood           | enum (radio)   | Настроение: Радостно / Спокойно / Грустно / Тревожно / Злюсь |
| energy_level   | number (1–10)  | Уровень энергии                           |
| tags           | string         | Теги через запятую (необязательное поле)  |
| description    | text           | Подробное описание дня                    |
| created_at     | datetime       | Дата и время создания записи (генерируется автоматически) |

Все требования к модели данных выполнены:
- 6+ полей;
- string — `title`, `tags`;
- date — `entry_date`;
- enum (checkbox/radio) — `mood`;
- text (длинный текст) — `description`.

## Структура проекта

```
lab6/
├── index.php       — главная страница с формой добавления записи
├── process.php     — обработка POST-данных, валидация, сохранение в JSON
├── view.php        — вывод всех записей в таблицу с сортировкой
├── style.css        — стили
└── data/
    └── entries.json — файл с сохранёнными данными
```

## Как запустить

1. Поместите папку `lab6` в директорию веб-сервера (например, через встроенный сервер PHP):
   ```
   php -S localhost:8000 -t lab6
   ```
2. Откройте в браузере `http://localhost:8000/index.php`.
3. Заполните форму и нажмите "Сохранить запись".
4. Перейдите на вкладку "Все записи", чтобы увидеть таблицу с возможностью сортировки по клику на заголовок столбца.

Папка `data/` должна быть доступна для записи (права на запись для веб-сервера).

## Как это работает

- **index.php** отображает HTML-форму. После неудачной отправки (ошибки валидации) пользователь возвращается сюда же — сообщения об ошибках и ранее введённые данные подгружаются из сессии, чтобы не перезаполнять форму заново.
- **process.php** принимает данные из `$_POST`, выполняет серверную валидацию (обязательные поля, длина строк, корректность даты, диапазон чисел, допустимые значения настроения), и либо сохраняет запись в `data/entries.json`, либо возвращает пользователя на форму с описанием ошибок.
- **view.php** читает все записи из JSON-файла и выводит их таблицей. Клик по заголовку столбца меняет параметр сортировки в URL (`?sort=поле&dir=asc|desc`).
- Все значения, выводимые в HTML, экранируются функцией `escape()` (обёртка над `htmlspecialchars`) для защиты от XSS.

## Ответы на контрольные вопросы

**1. Какие существуют методы отправки данных из формы на сервер? Какие методы поддерживает HTML-форма?**

HTML-атрибут `method` у тега `<form>` поддерживает два значения:
- **GET** — данные передаются в строке запроса (URL), видны пользователю, имеют ограничение по длине, подходят для поиска и фильтрации (некритичные, неконфиденциальные данные).
- **POST** — данные передаются в теле HTTP-запроса, не отображаются в URL, не имеют практического ограничения по размеру, подходят для отправки форм с конфиденциальными или объёмными данными (как в этой работе).

**2. Какие глобальные переменные используются для доступа к данным формы в PHP?**

- `$_GET` — данные, переданные методом GET (через query string).
- `$_POST` — данные, переданные методом POST (используется в данной работе).
- `$_REQUEST` — объединяет `$_GET`, `$_POST` и `$_COOKIE` (использовать с осторожностью, так как не всегда ясно, откуда пришли данные).
- `$_FILES` — данные о загруженных через форму файлах (при `enctype="multipart/form-data"`).

**3. Как обеспечить безопасность при обработке данных из формы (например, защититься от XSS)?**

- Экранировать все выводимые в HTML данные с помощью `htmlspecialchars()` (в проекте — функция `escape()`), чтобы спецсимволы (`<`, `>`, `"`, `'`) не интерпретировались браузером как HTML/JS-код.
- Выполнять серверную валидацию данных независимо от клиентской (атрибуты `required`, `minlength` и т.д. можно обойти, отправив запрос напрямую).
- Использовать `trim()` и проверки длины/формата перед сохранением данных.
- При работе с базами данных — использовать подготовленные выражения (prepared statements), чтобы избежать SQL-инъекций (в данной работе используется хранение в файле, поэтому это не требуется напрямую, но является хорошей практикой для будущих лабораторных).
- Ограничивать максимальную длину полей, чтобы избежать переполнения и DoS через слишком большие запросы.
- Не доверять данным из `$_POST`/`$_GET` напрямую — всегда проверять и фильтровать перед использованием.
